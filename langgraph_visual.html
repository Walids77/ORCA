🧠 What is LangGraph?

One page, drawn. Last updated: 2026-07-07 (Session 9)

1 · The one-line idea

A normal Python script runs top to bottom, one line after another — like a to-do list you can't reorder:

step 1
▼
step 2
▼
step 3

LangGraph lets you write the program as a flowchart that actually runs: boxes = jobs (nodes), arrows = what runs next (edges) — and arrows may branch, run side-by-side, or loop back:

decide
▼   ▼
path A
path B
▼   ▼
combine
💡 Why ORCA needs this: your 2D-decomposition design IS a flowchart — "decide simple-vs-hard, run sub-questions in parallel (breadth), some steps must wait for earlier answers (depth), then combine." A straight top-to-bottom script can't express that; a graph is exactly that.

2 · The smoke test you just ran — drawn

Two nodes, one shared notebook (LangGraph calls it "state") passed from box to box. Each box reads the notebook, does its one job, writes its result in.

START
▼
notebook: { text: "" }
▼
greetwrites a greeting
▼
notebook: { text: "hello from node 1" }
▼
shoutuppercases it, adds a stamp
▼
notebook: { text: "HELLO FROM NODE 1 → passed through node 2" }
▼
END
📓 The notebook is the whole trick. Nodes never talk to each other directly — they only read/write the shared notebook. That's how the future "combine" node will see everything the three retrieval legs found.
