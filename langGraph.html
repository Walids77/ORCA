🧭 ORCA's Brain — the Autonomous-Analyst design

The full target map, drawn. Logic only. Last updated: 2026-07-08 (Session 11). Companion to langgraph_visual.html (which explains what LangGraph is); this page is the design ORCA is being built toward.

1 · The shift — from "find" to "figure out"

Retrieval question — the answer already exists in a document.

"What is our leave policy?" → find the paragraph, return it. Plain RAG.

Analysis question — the answer exists nowhere; it must be computed and judged.

"Which department is overstaffed?" → pull the numbers, compute a ratio, compare to a benchmark, phrase a recommendation. The autonomous analyst.

💡 Why this matters: "chat with your documents" is crowded and unimpressive. "An agent that reads a company's data and produces a staffing / financial analysis with its reasoning shown" is differentiated — and it demonstrates the exact skills (tool use, multi-step reasoning, structured + unstructured data) the AI-engineer roles ask for. Validated demand: finance advisory + HR advisory are two of the most common agentic-RAG use cases right now.

2 · The four question levels → which path on the map

The standard Business-Intelligence ladder. It tells us which questions the map answers today (levels 1–2) and which earn a new branch later (levels 3–4).

🧩 Key rule (answers "can we add branches later?"): a new question shape is free — the router just writes a different plan over the same roads. Only a new capability (a forecast, a web search, an email connector) needs a new station. And because the map is fixed, adding one is clean.

Workers run side-by-side (breadth) or wait in order (depth — a step that needs an earlier answer). A worker can be a mini-chain: retrieve, then calculate.

🔒 Guardrail — the compute is caged. The agent never writes math. It calls pre-built, tested functions with parameters — like a calculator with fixed buttons, not a blank Python prompt. Keeps ORCA's security rule (never LLM-run raw code/SQL) and makes every computation eval-gradeable. Which functions we build is revealed by the 100+ test questions, then hard-coded.

5 · Zoom: the 3-layer answer — LLM as analyst, not just writer

So the LLM adds judgment without ever inventing a number. Worked on "Which department is overstaffed?" — Marketing: 12 staff, budget 12, on-budget; target 100, actual 78.

Layer 1 · THE NUMBERS (exact, from data) "Marketing: headcount 12, budget 12 — on budget. Target 100, actual 78."
Layer 2 · GROUNDED DIAGNOSIS (LLM connects the numbers — still traces to data) "Not overstaffed by headcount or budget — but 22% under target, so the salary spend isn't returning its cost."
Layer 3 · GENERAL INSIGHT (LLM's own experience — a labelled suggestion) "General suggestion, not from your data: a fully-staffed department under target more often has a skills / incentive gap than a headcount one — worth checking before cutting staff."
⚖️ Three rules keep it trustworthy: (1) Layer 3 never overrides Layers 1–2 — experience can suggest, never contradict the numbers. (2) Layer 3 is always a hypothesis to check, never a verdict — because this answer can get someone fired. (3) Its eval is different: Layers 1–2 grade against known answers; Layer 3 grades for "labelled? humble? doesn't contradict the data?". Decision support, not decision-making.

6 · One question, all the way through

"Which department is overstaffed?"

  1. Guard → Memory → Router. Router reads the catalog, sees a headcount table, a salary-budget table, a targets table → it's an analysis question → picks multi-step, writes the plan.
  2. Decompose writes the plan into the notebook — Step A (retrieve): headcount + salary budget per dept → SQL. Step B (calculate): spend vs budget → the variance function. Step C (depth, waits for B): for any dept not over budget, pull target vs actual → SQL, then the attainment-ratio function.
  3. Benchmark — "compared to what?" Here it's their own budgeted headcount + their own targets (source #1). If no target existed, ORCA asks you instead of guessing.
  4. Combine → the 3-layer answer (the Panel-5 example) — numbers, then grounded diagnosis, then the fenced suggestion.
  5. Self-check confirms Layers 1–2 are grounded and Layer 3 is labelled. Return with citations. You can then challenge it ("check the industry norm") and the conversation continues on memory.
🎯 The question came in as "overstaffed?" and ORCA gave back a better, more honest answer than the question — while never letting the LLM invent a number. That's the autonomous analyst.

7 · Two things this design also teaches

ORCA is a repeatable, tailorable service.

Every company has its own industry, its own data, its own definitions of words like "overstaffed". Building ORCA well = get sample files → pose 100+ questions (simple → analysis → report) → eval → fix → tailor. That loop is re-runnable per company. It's exactly the AI Solutions / Implementation Engineer job — and the discovery half (understand their business + data + needs) is where 15 years of client-facing experience is the edge, not a gap.

Build simple → complex, eval at each step.

The middle spine first (question → retrieval core → answer). Then the router. Then decompose with the retrieve-worker. Then the calculate-worker and the 3-layer answer last. Each addition must out-grade the simpler version on the eval, or it doesn't ship. The autonomous-analyst branch is the destination, not step one.
