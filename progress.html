🗺️ The plan (each layer proven before the next sits on it)

A

Give ORCA data — ingestion NEXT

Port the Excel extractor from old ORCA (per-file approval). One real Excel becomes three things on the PC — the food the brain eats:

📊 SQL rows → SQLite (exact numbers)
🧠 Vectors → ChromaDB (meaning/search)
📁 Stored copy of the file

Test: did the data arrive intact? (totals in SQL = totals in the Excel)

B

Build the brain — orchestrator on LangGraph LATER

Node by node: a router reads the question and picks a path — answer directly / one simple lookup / decompose into sub-questions (2D model: parallel branches + dependent steps) — then combines results into one answer with citations. Built in from day one: conversation memory + tracing (a log of every step, to find exactly where a wrong answer went wrong).

C

Prove it — terminal testing + eval harness LATER

No frontend yet: type a question in the terminal, read the answer. Grade with the eval harness — 5–10 questions with known answers, real numbers. Two grades: the result (right answer?) and the process (right database? how many steps? what cost?). Retrieval quality is graded on its own, first.

D

The doorway — local web frontend LATER

Login · upload / view / delete files · a prompt box that talks to the proven brain.

E

AWS — one service at a time LATER

Each piece added only when the app actually needs it, taught at that moment: S3 (files) · RDS Postgres + pgvector (replaces SQLite + ChromaDB in one DB) · Fargate (runs the backend) · Cognito (login) · Bedrock (Claude).

The principle behind the order: the logic decides the infrastructure. Prove the brain cheap and local first — AWS money and effort only go to something that already works. Data in → search works → brain reasons.
