Step 2 — Inside the Excel specialist. 7 stages turn one messy-tolerant spreadsheet into clean, accurate data.

1
Open & inventory
list the sheets, read each grid, trim empty trailing rows (built + tested ✓)
2
Find the table(s)
a sheet can hold more than one block — cut at blank rows, keep the main table, set side blocks (e.g. a "Fast Calculation" scratch box) aside by matching column types (built + tested ✓)
3
Read with types
dates=dates, numbers=numbers; "NA" and blank → proper empty (built + tested ✓)
4
Flag total rows
mark "Total / Average" rows so sums never double-count — proven: data rows re-summed to the sheet's own Total, exactly (built + tested ✓)
5
Understand columns
tag each column (meaning + aliases) + measure its real data type → "profit" later finds "Net Profit" (built + tested ✓)
6
Split into two
same table → exact-number rows + meaning chunks, each with a row fingerprint (hash) for incremental updates (built + tested ✓)
7
Store
write to all three stores below (built + tested ✓)

Step 3 — One file becomes three things (the food the brain eats):

📊 SQL rows ✓SQLite now → Postgres later. Typed table per sheet/PDF-table + tenant/citation/hash columns. Exact numbers proven (SUM matched the document's own total); re-upload doesn't duplicate; stale tables purged on re-ingest.
🧠 Vector chunks ✓Chroma + local model (all-MiniLM) now → Bedrock Titan V2 + pgvector later (one-line swap). Meaning-search returns the right rows + citations; re-upload doesn't duplicate.
📁 Raw file copy ✓local now → S3 later. Manifest + content hash → identical re-upload is skipped free.

✔️ Accuracy rule: the totals SQL computes must equal the totals printed in the document — else ingestion failed, loudly.

🔎 How a PDF turns into a searchable answer

Every uploaded PDF walks through 5 stages. Stages 1–4 happen once, at upload. Stage 5 happens each time you ask a question.

📄 PDF arrives
▼
1 · EXTRACT Docling reads the PDF into clean text + tables + page numbers · knows a heading from a paragraph
▼
2 · CHUNK cut the text into small passages · ~500 tokens each (≈350–400 words) · ~12% overlap · never crosses a heading
▼
3 · LABEL (metadata) stick a label card on every chunk: company · file · page · section · doc title
▼
4 · INDEX TWO WAYS each chunk gets a meaning-fingerprint and a keyword-index entry
▼
5 · SEARCH (hybrid) a question searches both ways, merges them, returns the top 5 passages

Stage 2 up close — the numbers you asked about. Each chunk is ~500 tokens (a token ≈ ¾ of a word), and each one repeats the last ~12% (~60 tokens) of the chunk before it.

CHUNK 1~500 tokens · ends with its last ~60 tokens →
the ~60-token
OVERLAPrepeated in both
CHUNK 2← starts with those same ~60 tokens · ~500 total

✔️ Why the overlap: if an important sentence sits right on a cut line, the repeat makes sure it lands whole inside at least one chunk instead of being sliced in half.

Stage 3 up close — the label card stuck on every chunk. This is what lets ORCA answer "…from page 33, section 10.3" and keep one company's data separate.

🏢 company + filetenant safety
📄 page"from page 33"
📑 section"§10.3 Corrective RAG"
📘 doc titleon every chunk
🔗 parent sectionhierarchy-ready

Light enrichment (today): we add a few helper words before fingerprinting — the authors chunk gets an "Authors (who wrote this)…" cue, captions get a "[Figure/Table caption]" tag, footnotes a "[Footnote, page 1]" tag. The heavy version (an AI summary on every chunk) needs Bedrock — parked.

Stage 5 up close — hybrid search (the engine behind the scoreboard above). Your question runs through both searches at once, then the two ranked lists get merged.

❓ your question
▼
🧠 MEANING searchembeddings · finds "money back" when you asked "refund"
🔡 KEYWORD search (BM25)exact words · finds "SKU-4471", "Baccarat", a section title · zero-word-overlap "hits" filtered out (Jul 6)
▼
🔀 MERGE the two lists — RRF, C = 5C decides how much being "#1 in one list" counts · low C = trust a strong single hit more (this rescued the metadata questions)
▼
🏆 top 5 passages → the answer material

✔️ Why both: meaning-search and keyword-search each catch what the other misses — running both and merging beats either alone. C = 5 validated on a 2nd unseen document (110-pg economics module, ~100%) → now the default. Jul-6 sharpening: zero-score keyword hits filtered + page-1 footnotes in their own small chunks → 87.5%.

Node by node: the brain now PLANS — a planner reads the question and the (permission-filtered) data catalog, writes a checklist (focused question · lane · waits-for), and a capped plan-runner executes it in parallel waves: each lap runs every step whose dependencies are ready; answers flow between waves. Walid's 2D model live: breadth = steps per wave · depth = waves. Built so far, each step eval-gated: the straight spine (Jul 9) → parallel legs (−28% time) → the ROUTER (13/13 lane picks, splits compound questions) → the PLANNER + WAVES, SHIPPED at 12/13 (beats the router's 11/13; the "best month → what did clients buy THAT month" dependency question passed in full for the first time) → plus the token/cost meter (every call: tokens + $ per answer) and the RBAC catalog fence (our script decides what each user's planner may even see — never the AI). LLM = Gemini gemini-2.5-flash for now → Claude via Bedrock once AWS unblocks it. Next (Session 16): the caged CALCULATE worker (exact arithmetic — average basket, % change, projections — by tested code, never AI math) + catalog meanings (one line per sheet so "what was bought" maps to client Sales, not company Expenses), then the hardening campaign: many eval rounds, no lucky passes.

C

Prove it — terminal testing + eval harness LATER

No frontend yet: type a question, read the answer. Grade with 5–10 known-answer questions. Two grades: the result (right answer?) and the process (right database? how many steps? what cost?).

D

The doorway — local web frontend LATER

Login · upload / view / delete files · a prompt box that talks to the proven brain.

E

AWS — one service at a time LATER

Each piece added only when the app needs it: S3 (files) · RDS Postgres + pgvector (replaces SQLite + Chroma in one DB) · Fargate (backend) · Cognito (login) · Bedrock (Claude).

The principle behind the order: the logic decides the infrastructure. Prove the brain cheap and local first — AWS money and effort only go to something that already works. Data in → search works → brain reasons.

Jul 9 · Session 13 numbers leg answers + total-row fix + 10/10 eval on the real Excel

The session that proved ORCA's differentiating half: exact numbers from a real business spreadsheet. The numbers leg used to only list which tables exist — now it answers, with a caged two-move design: move 1, the AI reads a catalog of the data (every sheet, its columns and types, one example row) and fills a small form — which sheet, which operation (SUM / AVG / MIN / MAX / COUNT / LIST), which column, which filters; move 2, our tested code checks every form field against the catalog, builds the database query itself, and always excludes Total rows. The AI never writes SQL. Then the eval: 10 known-answer figure-questions on the real retail Excel (4 Walid's, 6 Claude's incl. a trap). Run 1 scored 9/10 — but Walid challenged two "passing" figures and both were WRONG: an unlabelled grand-total row (a totals row with no "Total" text anywhere) had slipped past the keyword detector and sat in the store as data, silently doubling sums (average sale showed 75 instead of the true 37.58; pending sales 174 instead of 87). Even the eval's answer key had inherited the bug — the honest score was 7/10, and the permanent lesson is written down: an answer key must be computed independently of the pipeline it grades, and a figure that looks off must be chased even when the scoreboard is green. The fix used Walid's own idea — read the formula behind each cell: a second, formula-mode pass over the file flags any row computed by a vertical =SUM(...) down its own column, label or no label (with two refinements the proof-check forced: row-wise sums don't count, and sums pointing at another sheet don't count — the Summary's monthly rows are built from =SUM(Sales!…) and are data, not totals). Plus a shape fallback (a mostly-empty row whose lone number equals its column's sum) and the original label detector — union of all three. After the fix, every column with a printed total matches to the cent, and a second small fix (when a query matches nothing, the evidence now says "NO DATA" in words instead of a confusing None) made the trap refusal deterministic: final score 10/10. Bonus deliveries: a stage-6 data-quality scan that reports source-file problems with exact row numbers (found 14 real ones — year-1900 dates, text like "23 april" in a date column; agreed rule: ORCA stores face values and reports problems, the user fixes the source — ORCA never invents data or columns); the text leg now searches the whole corpus (a client-purchases question was answered complete-and-correct from free-text remarks, verified against the sheet); and brain_trips.html — three real trips drawn station by station. Next session (rehearse together first, then build, then eval): run the two legs in parallel, then eval the never-tested both-legs question (number + explanation in one answer), and only then decide the router on that evidence.

Jul 8 · Session 11 brain map drawn + the "autonomous analyst" upgrade

Design session, logic only — we drew ORCA's full brain map (saved as langGraph.html): guard → load memory → router (reads a catalog of the tenant's data + its agreed definitions) → three paths (answer directly / one lookup / decompose) → shared retrieval core → combine → self-check → answer, with the human able to challenge and the conversation continuing on memory. The big idea Walid drove: ORCA must answer analysis questions, not just retrieval questions. "What's the leave policy?" is a lookup; "which department is overstaffed?" is a conclusion that exists nowhere — it must be computed and judged. Three upgrades to the map followed: (1) the router's data catalog also holds the agreed benchmarks (what "overstaffed" means for this company); (2) the decompose branch gains a second worker type — CALCULATE — that is caged (it calls pre-built, tested functions like a calculator with fixed buttons, it never lets the AI write code, keeping the security rule and making every number checkable); (3) the answer comes back in three labelled layers — the exact numbers, the grounded diagnosis (the AI connecting those numbers, still traceable to data), and a clearly-fenced "general insight" (the AI's own experience, labelled "suggestion, not from your data", never allowed to override the numbers). We also mapped the four business-question levels (descriptive / diagnostic / predictive / prescriptive) onto the map — the first two are built, the last two are future branches — which answered Walid's question "can we add branches later?": a new question shape is free (the router just re-plans over the same roads); only a new capability (a forecast, web search) needs a new station. Bigger realization: ORCA is a repeatable, tailorable service — each company has its own data + definitions, so building it well (sample files → 100+ test questions → eval → fix → tailor) IS the AI Solutions/Implementation Engineer job, where Walid's client-facing background is the edge. Build order unchanged: the simple spine first, the analyst branch last, each step eval-gated.
