dates=dates, numbers=numbers; "NA" and blank → proper empty (built + tested ✓)

5

write to all three stores below (built + tested ✓)

Step 3 — One file becomes three things (the food the brain eats):

📊 SQL rows ✓SQLite now → Postgres later. Typed table per sheet/PDF-table + tenant/citation/hash columns. Exact numbers proven (SUM matched the document's own total); re-upload doesn't duplicate; stale tables purged on re-ingest.
🧠 Vector chunks ✓Chroma + local model (all-MiniLM) now → Bedrock Titan V2 + pgvector later (one-line swap). Meaning-search returns the right rows + citations; re-upload doesn't duplicate.
📁 Raw file copy ✓local now → S3 later. Manifest + content hash → identical re-upload is skipped free.

✔️ Accuracy rule: the totals SQL computes must equal the totals printed in the document — else ingestion failed, loudly.

🔎 How a PDF turns into a searchable answer

Every uploaded PDF walks through 5 stages. Stages 1–4 happen once, at upload. Stage 5 happens each time you ask a question.

📄 PDF arrives
▼
1 · EXTRACT Docling reads the PDF into clean text + tables + page numbers · knows a heading from a paragraph
▼
2 · CHUNK cut the text into small passages · ~500 tokens each (≈350–400 words) · ~12% overlap · never crosses a heading
▼
3 · LABEL (metadata) stick a label card on every chunk: company · file · page · section · doc title
▼
4 · INDEX TWO WAYS each chunk gets a meaning-fingerprint and a keyword-index entry
▼
5 · SEARCH (hybrid) a question searches both ways, merges them, returns the top 5 passages

Stage 2 up close — the numbers you asked about. Each chunk is ~500 tokens (a token ≈ ¾ of a word), and each one repeats the last ~12% (~60 tokens) of the chunk before it.

CHUNK 1~500 tokens · ends with its last ~60 tokens →
the ~60-token
OVERLAPrepeated in both
CHUNK 2← starts with those same ~60 tokens · ~500 total

✔️ Why the overlap: if an important sentence sits right on a cut line, the repeat makes sure it lands whole inside at least one chunk instead of being sliced in half.

Stage 3 up close — the label card stuck on every chunk. This is what lets ORCA answer "…from page 33, section 10.3" and keep one company's data separate.

🏢 company + filetenant safety
📄 page"from page 33"
📑 section"§10.3 Corrective RAG"
📘 doc titleon every chunk
🔗 parent sectionhierarchy-ready

Light enrichment (today): we add a few helper words before fingerprinting — the authors chunk gets an "Authors (who wrote this)…" cue, captions get a "[Figure/Table caption]" tag, footnotes a "[Footnote, page 1]" tag. The heavy version (an AI summary on every chunk) needs Bedrock — parked.

Stage 5 up close — hybrid search (the engine behind the scoreboard above). Your question runs through both searches at once, then the two ranked lists get merged.

❓ your question
▼
🧠 MEANING searchembeddings · finds "money back" when you asked "refund"
🔡 KEYWORD search (BM25)exact words · finds "SKU-4471", "Baccarat", a section title · zero-word-overlap "hits" filtered out (Jul 6)
▼
🔀 MERGE the two lists — RRF, C = 5C decides how much being "#1 in one list" counts · low C = trust a strong single hit more (this rescued the metadata questions)
▼
🏆 top 5 passages → the answer material

✔️ Why both: meaning-search and keyword-search each catch what the other misses — running both and merging beats either alone. C = 5 validated on a 2nd unseen document (110-pg economics module, ~100%) → now the default. Jul-6 sharpening: zero-score keyword hits filtered + page-1 footnotes in their own small chunks → 87.5%.

B

Build the brain — orchestrator on LangGraph NEXT

Node by node: a router reads the question and picks a path — answer directly / one simple lookup / decompose into sub-questions (2D model: parallel branches + dependent steps) — then combines results into one answer with citations. Built in from day one: conversation memory + tracing.

C

Prove it — terminal testing + eval harness LATER

No frontend yet: type a question, read the answer. Grade with 5–10 known-answer questions. Two grades: the result (right answer?) and the process (right database? how many steps? what cost?).

D

The doorway — local web frontend LATER

Login · upload / view / delete files · a prompt box that talks to the proven brain.

E

AWS — one service at a time LATER

Each piece added only when the app needs it: S3 (files) · RDS Postgres + pgvector (replaces SQLite + Chroma in one DB) · Fargate (backend) · Cognito (login) · Bedrock (Claude).

The principle behind the order: the logic decides the infrastructure. Prove the brain cheap and local first — AWS money and effort only go to something that already works. Data in → search works → brain reasons.
